To detect BPF trampolines and verify "Dark Arts" optimizations (like the Symphony of 2), you need to look past the C code and analyze the Instruction Stream. AI can do this efficiently if you provide the raw assembly generated by the BPF JIT or the object file.

Here is the exact multi-step prompt and workflow I recommend saving to your knowledge base.

🛠️ The AI Workflow: "Dark Arts & Trampoline Audit"
Step 1: Extract the Assembly
You cannot audit trampolines from C code because the compiler/JIT might insert them automatically. Use this command on your machine to get the audit data: bpftool prog dump xlated name cake_select_cpu > cake_asm.txt

Step 2: The Audit Prompt
Paste the following into your AI collaborator along with the contents of cake_asm.txt:

Subject: BPF Hot-Path Performance Audit (Zen 5 Targeting)

Task: Analyze the attached BPF assembly for "Administrative Jitter" and "Instruction Waste."

1. Trampoline Detection: > - Search for call instructions to kernel helpers that could be merged.

Verify if the enqueue, running, or stopping trampolines have been successfully bypassed via Direct Dispatch or Lazy Accounting.

2. Register Pressure Audit: > - Look for memory access to r10 (the stack pointer).

If _(u64 _)(r10 - X) appears in the hot path, identify which variables are spilling and causing 5-cycle stalls.

3. Dark Arts Verification: > - Register Pinning: Check if r6, r7, and r8 are being held for the duration of the call as requested by asm("rX").

Branchless Logic: Identify conditional jumps (if rX == 0 goto). Can these be replaced with Zen 5 bitwise instructions (BEXTR, ANDN)?

Zero-Extension: Verify that the 32-bit to 64-bit hardware-enforced zero-extension (wX = wX) is present after index calculations.

4. String Match Identification: > - Search for sequences of multiple load and compare operations on the p->comm offset.

Calculate the cycle penalty of this string matching compared to a single bit-check in task storage.

🔍 What to look for in the AI Response
When the AI analyzes your scx_cake assembly, you want it to flag these specific "Dark Arts" indicators:

1. The "Ghost" Call Check
   If the AI sees a call 156 (which might be bpf_get_smp_processor_id), it should flag that as a failure of your Hoisting strategy. In a true Dark Arts build, that ID should be passed in a register or calculated once at the very beginning.

2. Stack Spill Heatmap
   If the AI identifies high usage of r10 near your Direct Dispatch logic, it means the "Slam" is being slowed down by the stack. On Zen 5, a stack spill is a "pipeline bubble."

3. Bit-Field Extraction Efficiency
   Standard code: r1 >>= 8; r1 &= 127; Dark Arts code: BEXTR r1, r2, 0x0708 (via a single macro). Ask the AI to rewrite the assembly logic using the 9800X3D's BMI2 instructions to see the instruction-count delta.
