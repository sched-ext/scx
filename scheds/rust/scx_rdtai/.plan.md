# scx_rdtai: Optimization & Fixes Plan

## 1. Zero-Copy Reward Aggregation
**Current State:** BPF performs a `bpf_map_lookup_elem` and an atomic add for every task exit to update the global `q_rewards` map. This introduces measurable overhead in micro-benchmarks like Hackbench and Redis.
**Optimization:** 
- Convert `q_rewards` into a `BPF_MAP_TYPE_PERCPU_ARRAY`.
- BPF will update its local CPU's slot (no atomic operations needed).
- Rust will iterate over all CPUs and aggregate the rewards once per second during the `optimize_q_table` loop.
- **Expected Impact:** Fixes the IPC/Messaging performance gap vs FIFO.

## 2. Epsilon-Greedy "Interactive" Optimization
**Problem:** The 5% exploration rate (Epsilon) can randomly delay time-sensitive UI or interactive threads, causing slightly higher P50/P90 latencies in Redis.
**Solution:**
- Modify BPF to check `taskc->avg_runtime` before deciding to explore.
- If the task is categorized as "Interactive" (Bucket 0/1), set `epsilon = 0` for that specific decision.
- **Expected Impact:** Blazing fast FIFO-like speeds for UI/interactive tasks while retaining AI learning for complex/compute workloads.

## 3. Fix the Jitter Report
**Problem:** `sch_tests/run_benchmark_lat.sh` fails to parse `cyclictest` histograms correctly on this system, resulting in `-nan` or incorrect values.
**Fix:**
- Re-write the `get_cyclictest_percentile` function to handle the standard multi-CPU histogram format correctly.
- Ensure the Avg/Max parsing correctly identifies the highest values across all cores.
- **Expected Impact:** Accurate visibility into the scheduler's HFT (High Frequency Trading) performance.
