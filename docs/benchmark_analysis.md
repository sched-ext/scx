# SCX_RDTAI: centrales Benchmark Analysis & Performance Report

This document compiles, reviews, and analyzes the performance metrics of the **`scx_rdtai`** (Real-time Decision Tree AI) scheduler compared to other `sched_ext` schedulers (`scx_rlfifo`, `scx_rusty`, and `scx_rustland`). 

Data is synthesized from the central throughput benchmarks, Redis database evaluations, network throughput simulations, and real-time wakejitter testing.

## 2. Key Advantages (Where `scx_rdtai` Excels)

* **Outstanding Tail Request Latency (Schbench):**
  Under extreme thread saturation, `scx_rdtai` completes requests much faster than other schedulers. At the **99.9th percentile**, it finishes request processing in **`20.9ms`** (more than **2x to 3x faster** than other schedulers, which take `45ms` to `64ms`).
* **High Network Throughput (`iperf3`):**
  Reaches a maximum of **`121.24 Gbps`** on network transfers, yielding an ~18% advantage over `scx_rustland` and a ~55% increase over `scx_rusty`. This is due to its **Action 2 (Run Now)** logic, which fast-paths network socket/interrupt wakeups.
* **Superb Median Wakeup Latencies:**
  At the 50.0th percentile (under normal conditions), tasks wake up and begin executing in just **`221us`**—beating out `rusty` (`486us`) and `rustland` (`851us`).
* **Efficient Build Compilation Parallelization:**
  Ties with `rustland` for the fastest Linux Kernel compilation speed (**`106.42s`**), beating `rusty` by about 1 second.

---

## 3. Disadvantages & Architectural Bottlenecks

* **Poor Cache Locality (20.71% Cache Misses):**
  `scx_rdtai` incurs a high L3 cache miss rate of **`20.71%`** (nearly double that of `rlfifo` at `13.45%`). The Decision Tree's migration actions (**Action 1**) aggressively load-balance threads across domains, which frequently disrupts CPU cache lines.
* **High Tail Wakeup Jitter under Saturation:**
  Under high contention, its wakeup tail latency jumps to **`7.96ms`** (P99) and **`12.3ms`** (P99.9). This is because the scheduler occasionally delays minor background threads to keep high-reward interactive threads running.
* **Microsecond-Scale Overhead for Light DB Operations:**
  For ultra-fast key-value lookups (Redis GET/SET), `scx_rdtai` has a higher P50 median latency (`0.21ms` vs `0.07ms` for `rustland`). This is the result of the overhead generated by BPF collecting PMU metrics and evaluating the Decision Tree for every fast context switch.

## 4. Workload Suitability Matrix

| Workload Class | Suitability | Rationale |
| :--- | :---: | :--- |
| **I/O & Network Services** (e.g., Nginx, gRPC servers, Kafka) | **Excellent** | Benefits directly from `121 Gbps` network throughput and quick socket wakeup loops. |
| **Interactive Applications** (e.g., UI, desktop, gaming) | **Excellent** | Eliminates long tail pauses, keeping request completions under 21ms at P99.9. |
| **Heavy Batch Compilations** (e.g., `make -j$(nproc)`) | **Excellent** | Effectively saturates all CPU cores with minimal userspace scheduling latency. |
| **Microservices / Ultra-Low Latency KV Stores** | **Average** | Small decision overhead in BPF affects median Redis GET/SET operations. |
| **Cache-Sensitive Databases & HPC** | **Poor** | High cache miss rate (`20.71%`) bottlenecks memory/cache-sensitive execution loops. |

---

## 5. Actionable Optimization Targets

To further refine `scx_rdtai`'s behavior, focus on:
1. **Damping Action 1 (Migrate):** Refine the threshold for migrating tasks to restrict CPU-hopping when a task's cache miss rate is high.
2. **Smoothing the Tail Wakeup Jitter:** Introduce a maximum waiting time cap in BPF that overrides the decision tree and immediately schedules tasks that have been starved for more than `2ms`.
